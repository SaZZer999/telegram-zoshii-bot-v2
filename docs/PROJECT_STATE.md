# PROJECT_STATE.md

Короткий знімок поточного стану проєкту — для швидкого старту нової сесії Claude Code без повторного розкопування історії чату. Детальний опис фіч і плани — `docs/PROJECT.md` / `docs/ROADMAP.md`.

## Поточна архітектура

* **Flask webhook** (`bot.py`) — один процес, синхронна обробка кожного Telegram-запиту в межах одного HTTP-виклику.
* **PostgreSQL / Supabase** через `psycopg` (`database.py`) — постійне сховище, ідемпотентні міграції при старті (`init_db()`).
* **Gemini API** — весь AI-розбір тексту й chat, прямі HTTP-запити через `requests` (без SDK).
* **Groq Whisper** — транскрипція голосових повідомлень (`voice_input.py`).
* **Render Web Service** — хостинг, деплой автоматично при push у `main`; процес запускається через `gunicorn`, не через `python bot.py` (dev-режим лишився тільки для локального запуску).

## Важливі файли

* `bot.py` — Flask-маршрути (`/`, `/health`, `/webhook/<TELEGRAM_BOT_TOKEN>`), клавіатури, промпти, pending-стан у пам'яті процесу.
* `message_dispatcher.py` — впорядкований роутинг вхідного тексту (навігація → спецкнопки → меню → pending-стани → команди → AI-фолбек).
* `database.py` — весь доступ до PostgreSQL, транзакції, stale-snapshot захист.
* `expenses.py` — домен витрат.
* `household_router.py` — Global Household Router v1 (змішані побутові команди одним Gemini-викликом).
* `mini_action_planner.py` — Unified Mini Action Planner V1: окремий, вузький last-resort Gemini-класифікатор (`add_to_shopping`/`add_to_inventory`/`ask_inventory`/`meal_ideas`/`unknown`), спрацьовує останнім у Phase D перед загальним AI-чатом. **Не змінювався** цією задачею.
* `action_planner.py` — Inventory Action Planner V1 (деталі нижче) — окремий від `mini_action_planner.py` модуль, інша дія, інше місце в routing.
* `shopping_action_planner.py` — Shopping Action Planner V1 (деталі нижче) — окремий від `action_planner.py`/`mini_action_planner.py` модуль, лише `shopping_delete`/`shopping_mark_bought`, спрацьовує поза shopping mode/saved-list context.
* `preview_editing.py` — деталі нижче ("Preview Edit").
* `quantities.py` — парсинг/форматування структурованих кількостей (одна спільна реалізація для `bot.py`/`database.py`).
* `voice_input.py` — голосовий ввід (Groq Whisper).
* `photo_receipts.py` — розпізнавання чеків з фото.
* `Procfile`, `requirements.txt` — деплой на Render (gunicorn).
* `docs/PROJECT.md` — повний опис фіч. `docs/ROADMAP.md` — план розвитку. `docs/DEPLOYMENT_RUNBOOK.md` — інструкція деплою/troubleshooting.

## Expense Batch Delete V1 — не перевірено наживо (2026-07-15)

* **Live-баг:** реальний список витрат мав `Тест кава batch — 51,23 zł` і `Тест еспресо batch — 14,00 zł` (`Тест чай batch` НЕ існував як витрата — окремий баг контексту раніше додав його до покупок як `52,37 шт`, див. Команди «Видали тест чай batch, тест кава batch і тест еспресо batch» і «Видали тест кава batch і тест еспресо batch» не створювали batch preview — замість цього бот показував «Не зміг однозначно визначити витрату» з ПОВНИМ нефільтрованим списком останніх витрат (комод, дитяче ліжечко, подарунок доньці — жодне з них не стосувалося команди).
* **Точна причина:** existing expense-delete схема й pending-стан були розраховані лише на ОДНУ ціль; кілька названих витрат стискалися в single-selection ambiguity, після чого fallback показував top-N recent expenses без relevance filtering.
* **Structured schema:** `EXPENSE_ROUTER_PROMPT`'s (`expenses.py`) delete_expense-відповідь отримала нове поле `targets` — масив 1–10 об'єктів `{"description", "amount", "date_hint"}`, витягнутих Gemini БЕЗПОСЕРЕДНЬО з тексту повідомлення (кома/«і»/«та»/«а також» між назвами — окремі елементи); `selected_numbers` лишився для bare-номер відповідей у режимі вибору зі списку. Gemini НІКОЛИ не повертає DB id — усе резолвиться в Python (`expenses._resolve_expense_targets`/`_resolve_single_target`) проти живих витрат (`get_recent_expenses_for_deletion`). Це й далі ОДИН Gemini-виклик на повідомлення (`_ask_gemini_expense_router` не змінив свій call-count).
* **Resolution (all-or-nothing):** кожна target резолвиться незалежно — exact canonical-match першим пріоритетом, потім relevance-filtered token-overlap (`_expense_query_tokens`/`_expense_candidate_relevant`, з мінімальним порогом релевантності: власний токен опису АБО сума АБО дата мають збігтися). Один expense row ніколи не використовується двічі в одному batch (резолвлений row вилучається з пулу для наступних targets). Якщо ХОЧА Б ОДНА target відсутня — жодного preview, жодного часткового видалення; повідомлення прямо називає, чого не знайдено, і окремо перелічує знайдені (але не видалені) позиції. Якщо якась target неоднозначна — показуються ЛИШЕ релевантні candidates САМЕ для цієї target (ніколи повний список останніх витрат).
* **Existing single-delete fallback також виправлено:** `_resolve_expense_delete_selection`'s ambiguous-гілка (номерований режим вибору й глобальна команда без структурованих targets) тепер теж relevance-filtered замість показу повного `recent_expenses` — 0 релевантних candidates → «Не знайшов витрати, схожої на «...».» замість повного списку. **Свідома зміна поведінки existing теста** — `tests/test_expense_delete.py::test_no_matching_candidate_returns_controlled_message` оновлено (стара версія асертувала показ повного нефільтрованого списку як правильну поведінку — той самий підхід, що й заміна `test_add_expense_writes_no_journal` у `cc50495`).
* **All-or-nothing preview:** рівно один резолвлений target і далі йде через existing `pending_expense_delete`/`_build_delete_preview_from_match` (без регресії); 2+ резолвлених targets створюють НОВИЙ `pending_expense_batch_delete` (той самий формат preview/keyboard: «✅ Так, видалити» / «❌ Скасувати»).
* **Atomic transaction:** новий `database.delete_expenses_batch`/`_verify_expense_targets_in_tx` — ОДНА транзакція, кожен target row перевіряється й лочиться (`FOR UPDATE`) проти снепшоту (amount/category/expense_date/description), один stale/змінений/відсутній row → `StaleSnapshotError` → rollback усього batch, жодного часткового видалення. Той самий патерн, що й `delete_items_batch`/`delete_expense`. DB schema НЕ змінена.
* **Pending state:** `pending_expense_batch_delete` (chat_id -> `{household_id, expense_ids, snapshots, total, origin}`) — додано до `interaction_state.py`'s blocking-груп (`_expense_gate_blocking_states`/`_active_expense_preview_states`) і до confirm/cancel routing у `bot.py`, так само, як `pending_expense_delete`/`expense_delete_selection`. Confirm/cancel і далі мають найвищий пріоритет; повторний confirm після успішного видалення показує «Немає активної дії для підтвердження.» (pending-стан вилучається ДО DB-виклику).
* **Undo НЕ реалізовано в цій задачі:** `database.delete_expense`/`delete_expenses_batch` і далі не пишуть у `household_action_journal` — «↩️ Скасувати останню дію» так само не підтримує жодне видалення витрат, ні одиночне, ні batch (попередньо існуючий стан, задокументований у розділі «Expense-delete natural-language routing» вище; свідомо поза межами цієї задачі).

## Quantity + Price Intent Clarification V1 — не перевірено наживо (2026-07-16)

* **Мета:** `товар + кількість + ціна` в одному повідомленні («Молоко 1 л 4,99 zł») більше не вимагає ручного розбиття на дві команди. Active List Context Routing Stabilization V1 (`d885160`)/Context Intent Safety V1 (`6054fe2`) вже безпечно блокували цю неоднозначність, але лише статичною відмовою («напиши окремо…») без жодного способу продовжити з того самого повідомлення — ця задача перетворює ту відмову на дієве уточнення з чотирма варіантами.
* **Уточнення:** без явного дієслова (купівлі, «Додай», «Запиши») бот показує `pending_quantity_price_intent` — новий, вузький, RAM-only pending-стан із чотирма кнопками: «🛒 Додати до покупок», «💸 Записати витрату», «✅ Уже купив», «❌ Скасувати». Стан зберігає лише вже детерміновано розібрані дані (item_name/quantity_value/quantity_unit/quantity_text/amount/household_id/user_db_id/origin) — без жодного Gemini-виклику для самого тригера (`quantities._QUANTITY_UNIT_RE`/`parse_structured_quantity` для кількості, `expenses._EXPENSE_AMOUNT_RE`/`_parse_expense_amount` для суми — ті самі regex/parser, що вже використовує `looks_like_money_amount`/`looks_like_explicit_item_quantity`, лише тепер для ізоляції обох фрагментів замість самого лише виявлення).
* **Кожен вибір:**
  * **«🛒 Додати до покупок»** — `household_router.build_add_preview_from_items("add_shopping", ...)` → ТОЙ САМИЙ `pending_global_household`/confirm/`apply_global_household_operations` шлях, що вже використовують Global Explicit/Bare Add. Ціна ніколи не перетворюється на кількість і ніде не зберігається (без зміни схеми) — прев'ю чесно показує нотатку, що ціну не записано.
  * **«💸 Записати витрату»** — `expenses.build_receipt_expense_preview(...)` (ТОЙ САМИЙ `pending_expense`/confirm/`add_expense` шлях, що вже використовує Photo Receipt Input V1 для попередньо валідованих даних). Товар до покупок/запасів не додається. Категорія — «Інше» (не вгадується глибше без Gemini-класифікації), з тим самим «(не вдалося визначити точно)» позначенням, що й інші auto-defaulted категорії.
  * **«✅ Уже купив»** — reuse ІСНУЮЧОГО compound-executor'а: `household_router.build_add_preview_from_items("add_inventory", ...)` (ТІ САМІ Inventory Representation Guard перевірки, що вже проходять Global Explicit/Bare Add) з подальшим встановленням `new_expenses`/`new_expense` на ТОМУ САМОМУ payload перед `_handle_household_router_result` — ОДНЕ комбіноване `pending_global_household` прев'ю, застосовується атомарно ТОЄЮ САМОЮ транзакцією `apply_global_household_operations`, яку вже використовує «Купив X за Y zł». Жодного нового DB-executor'а. Якщо Representation Guard не може безпечно об'єднати (`kind != "ok"`) — контрольоване повідомлення, **нічого не пишеться** (ні товар, ні витрата окремо).
  * **ВІДОМЕ ОБМЕЖЕННЯ (задокументовано, не реалізовано цією задачею):** «✅ Уже купив» не шукає й не позначає/не прибирає відповідну позицію зі списку покупок — жодна існуюча функція в проєкті цього не робить автоматично, а створення такої логіки вимагало б ширшого refactor поза фокусом цього фіксу. Прев'ю чесно обіцяє лише те, що виконує: додавання до запасів + витрату, нічого про список покупок.
  * **«❌ Скасувати»** — той самий спільний cancel-handler (giant `elif chat_id in pending_XXX` chain у `bot.py`), нова гілка `elif chat_id in pending_quantity_price_intent`; той самий undo-button-cancels-active-operation guard (`_has_active_pending_clarification_or_preview`/`_cancel_active_pending_operation`).
* **Explicit verbs обходять уточнення:** дієслово купівлі («купив»/«придбав»/**«взяли»/«взяв»** — останні два додано до `household_router._BOUGHT_RE` і синхронізовано в `legacy_shopping_flow._PURCHASE_VERB_RE`/`legacy_inventory_flow._PURCHASE_VERB_RE`, той самий root вже присутній у `shopping_action_planner._BOUGHT_OR_TAKEN_VERB_ROOTS`) веде прямо в existing Global Household Router compound-flow (ОДИН Gemini-виклик, як і раніше). Явний префікс «Додай»/«Додайте»/«Додати» (`_AMBIGUOUS_ADD_PREFIX_RE`) і явний префікс «Запиши…» (новий `_EXPLICIT_EXPENSE_VERB_RE`) також обходять уточнення — повідомлення з такими маркерами взагалі не проходить через новий route.
* **Позиція в routing:** новий global route `_route_quantity_price_clarification` — окреме поле `CommandRouteDeps.quantity_price_clarification_route` у `message_dispatcher.py`, перевіряється ОДРАЗУ після `active_list_context_route` і ПЕРЕД `ambiguous_add_route`/`explicit_global_add`/`bare_global_add`/`global_household_router`/`global_expense_command` — так money+quantity ambiguity перехоплюється РАНІШЕ, ніж `global_expense_command`'s домен-сліпий zł-гейт встиг би мовчки створити expense-only прев'ю. Всередині активного `saved_list_context`, `_route_active_list_context_command`'s власна money+quantity гілка (`d885160`) вже будує ТЕ САМЕ уточнення першою — новий global route спрацьовує лише коли жоден saved-list context не активний (головне меню).
* **Захист від двох одночасних pending states:** `pending_quantity_price_intent` додано до `interaction_state.InteractionStateDeps`, до `_alias_gate_blocking_states` (блокує всі інші gate-и), до `clear_interaction_state` (очищається при навігації), до нового pending-route слота в `message_dispatcher._dispatch_pending_routes` (`PendingRouteDeps.pending_quantity_price_intent`/`continue_quantity_price_intent`) — поки уточнення активне, БУДЬ-який текст (крім confirm/cancel, вже перевірених раніше в V3B) резолвиться лише проти цього стану, ніколи не запускає новий route. Повторний вибір після pop() не виконує дію двічі — pending-стан вилучається ДО побудови наступного прев'ю (той самий "pop перед DB-викликом" патерн, що й в усіх інших confirm-handler'ах).
* Змінені файли: `bot.py` (новий `pending_quantity_price_intent` dict, секція «QUANTITY + PRICE INTENT CLARIFICATION V1» — `_extract_quantity_price_intent_fields`/`_start_quantity_price_clarification`/`_continue_quantity_price_intent`/три `_apply_quantity_price_choice_*`/`_route_quantity_price_clarification`, оновлена money+quantity гілка `_route_active_list_context_command`, нова cancel-гілка, розширені `_has_active_pending_clarification_or_preview`/`_cancel_active_pending_operation`, wiring у `_interaction_state_deps`/`_pending_route_deps`/`_command_route_deps`), `message_dispatcher.py` (нові поля `PendingRouteDeps.pending_quantity_price_intent`/`continue_quantity_price_intent`, `CommandRouteDeps.quantity_price_clarification_route`, новий check у `_dispatch_pending_routes`/`_dispatch_command_routes`), `interaction_state.py` (нове поле `InteractionStateDeps.pending_quantity_price_intent`, додано до `_alias_gate_blocking_states`/`clear_interaction_state`), `household_router.py` (`_BOUGHT_RE` розширено «взял\w*|взяв»), `legacy_shopping_flow.py`/`legacy_inventory_flow.py` (`_PURCHASE_VERB_RE` синхронно розширено), новий `tests/test_quantity_price_intent_clarification.py` (24 наскрізні webhook-тести), одна оновлена fake-deps фікстура в `tests/test_interaction_state_module.py`, дві оновлені асерції в `tests/test_active_list_context_routing.py` (свідома зміна поведінки — старий refusal замінено новим уточненням).

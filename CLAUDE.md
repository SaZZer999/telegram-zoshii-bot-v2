# CLAUDE.md

## Проєкт

Це приватний Telegram-помічник для одного домашнього господарства: Максима та його дружини.

Це не публічний застосунок, не SaaS-продукт і не комерційний сервіс. Основна мета — швидко створити корисного бота для щоденних побутових задач.

Детальний опис: `docs/PROJECT.md`
План розвитку: `docs/ROADMAP.md`

## Поточний стек

* Python
* Flask
* Telegram Bot API через webhook
* Gemini API (весь реальний AI-розбір і chat — прямі HTTP-запити через `requests`, без SDK)
* PostgreSQL (Supabase) через `psycopg`
* Render Web Service
* GitHub
* змінні середовища в `.env` локально та у Render Environment Variables

## Поточний стан

* Точка входу: `bot.py` — Flask webhook, головний Telegram-диспетчер (кнопки, режими введення, preview/confirm/cancel), підключає `expenses.py` та `household_router.py`.
* `database.py` **не порожній** — повний шар доступу до PostgreSQL: ідемпотентні міграції (`init_db`), CRUD для покупок/запасів/aliases/витрат, атомарні транзакції зі stale-snapshot захистом (`_verify_targets_in_tx`, `apply_*_operations`).
* `expenses.py` — домен витрат: додавання витрати (сума в zł, категорія, опис, дата), останні витрати, місячний підсумок за категоріями, видалення однієї витрати (з підтвердженням).
* `household_router.py` — Global Household Router v1: один Gemini-виклик розпізнає до п'яти типів побутових дій в одному повідомленні (додати до покупок, додати до запасів, часткове списання запасів, додати витрату, видалити витрату) незалежно від відкритого меню, і будує один спільний preview для змішаних команд (напр. «Купив масло за 10 zł» → запаси + витрата одним підтвердженням).
* Telegram-кнопки й reply-клавіатури реалізовані для головного меню, покупок, запасів, aliases і витрат.
* Пам'ять звичайного AI-чату (поза контекстом покупок/запасів/витрат) зберігається лише в RAM і зникає після restart.
* `requirements.txt` містить: flask, groq, python-dotenv, requests, psycopg[binary]. Пакет `groq` встановлений, але наразі не використовується для реальних відповідей — весь chat/parsing іде через Gemini.

## Архітектура файлів (де що шукати)

* `bot.py` — лише Flask/Telegram-диспетчер: маршрутизація тексту, кнопки, клавіатури, preview shopping/inventory.
* `expenses.py` — усе про витрати (додавання, звіти, видалення).
* `household_router.py` — глобальний менюнезалежний router для змішаних побутових команд.
* `database.py` — уся робота з PostgreSQL і транзакціями.
* Головне правило проєкту: **Gemini лише інтерпретує текст, Python перевіряє результат, PostgreSQL зберігає, користувач підтверджує кнопкою.**

## Обов'язкові правила

1. Спілкуйся зі мною українською простою зрозумілою мовою.
2. Перед нетривіальними змінами коротко пояснюй:

   * що плануєш зробити;
   * які файли зміняться;
   * можливі ризики;
   * як я зможу перевірити результат вручну.
3. Не змінюй архітектуру Flask webhook на polling без мого окремого дозволу.
4. Не додавай нові бібліотеки, не змінюй залежності та не встановлюй пакети без мого дозволу.
5. Не створюй, не видаляй і не змінюй схему бази даних без мого дозволу.
6. Не роби commit, push, deploy, reset, delete, rename або інші потенційно небезпечні операції без мого явного дозволу.
7. Ніколи не читай, не показуй, не редагуй і не коміть `.env`.
8. Ніколи не показуй токени, API-ключі, паролі, webhook URL із секретами або інші конфіденційні дані.
9. Надавай перевагу малим, безпечним і зворотним змінам замість переписування всього проєкту.
10. Після зміни коду пояснюй, що саме змінилося, і давай короткі кроки ручного тестування.
11. Якщо завдання нечітке або має кілька варіантів реалізації, спочатку постав короткі уточнювальні питання або запропонуй варіанти.
12. Не вважай функцію реалізованою, поки її не можна перевірити вручну.

## Дані та база

* Production-база вже підключена: PostgreSQL (Supabase) через `psycopg`, уся логіка в `database.py`. SQLite у проєкті не використовується.
* Схема й ідемпотентні міграції застосовуються автоматично при старті (`init_db()`); нових таблиць чи стовпців без потреби не додавай.
* Не змінюй схему бази, не додавай нові таблиці й не запускай ручні міграції без окремого затвердженого плану.

## Стиль роботи

* Не вигадуй функції, яких немає в `docs/PROJECT.md` або `docs/ROADMAP.md`.
* Не додавай "на майбутнє" складну архітектуру без потреби.
* Спочатку аналізуй поточний код, потім пропонуй мінімальний план.
* Працюй над однією невеликою функцією за раз.

## Робочий процес

Для невеликих і чітко сформульованих задач не роби окремий довгий план перед змінами.

Замість цього працюй так:

1. Знайди й прочитай лише те, що потрібно для конкретної задачі (Grep/Glob по релевантних функціях/файлах) — не читай `bot.py` повністю і не запускай subagent-ів заради невеликої, чітко сформульованої задачі.
2. Внеси мінімальні зміни.
3. Одразу запусти відповідну локальну перевірку.
4. Переглянь `git diff`.
5. Коротко поясни, що змінилося і як це перевірити вручну.

Не проси окремого підтвердження між редагуванням, перевіркою та показом diff.

Перед початком роботи все ще проси підтвердження лише для:
- нових бібліотек або зміни `requirements.txt`;
- бази даних, таблиць або міграцій;
- `.env`, API-ключів, токенів або webhook;
- Render, deploy або зміни хмарних налаштувань;
- видалення файлів;
- `git commit`, `git push`, `git reset`, force push;
- великих змін архітектури.

Для задач, що торкаються бази даних, транзакцій, preview/confirm-логіки або безпеки (доступ, секрети, валідація Gemini-результатів) — перевіряй суворіше: явно проглянь stale-snapshot захист, транзакційність (усе або нічого) і що preview справді нічого не змінює до підтвердження, перш ніж вважати задачу готовою.

Перед завершенням задачі сам перевір, що змінилися лише очікувані файли.
